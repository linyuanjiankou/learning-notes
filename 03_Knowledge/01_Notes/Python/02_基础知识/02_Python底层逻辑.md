# 缩进
由于C语言通过花括号`{}`和关键字来界定，缩进只是一种美观手段。但在Python当中，缩进不仅仅是一种风格，更是语法本身。

当你写下 `if x>0:` 时，Python解释器的词法分析器会在后台生成特殊的 **INDENT** 和 **DEDENT** 标记。这就像是在搭建积木，缩进决定了哪块积木属于哪一层。如果你混用了Tab和空格，或者缩进层级错乱，解释器就会抛出 `IndentationError` 。这并非简单的格式警告，而是语法错误，意味着代码块在逻辑上已经崩塌。

遵循PEP8规范，Python使用4个空格作为缩进单位，不仅是社区共识，更是为了避免跨编辑器协作时的隐形陷阱。
# 代码块
Python程序由模块、函数体、类定义等代码块构成。每一个代码块都是一个独立的执行单元，同游自己的命名空间。

例如，在函数 `defgreet(name):` 内部定义的变量就属于局部命名空间。一旦函数停止运行，这个空间就会销毁。这种机制保证了变量作用域的清晰，避免了全局变量的污染。同时，Python允许嵌套的代码块，比如在循环中嵌套条件判断，每一层缩进都代表一个新的逻辑层级，像俄罗斯套娃一样层层包裹，逻辑严密。
# 命名与绑定
Python中的一切皆对象，而变量只是对象的引用。

当你执行 `a=1` 时，Python并没有把1“装进”a盒子里，而是创建了一个整数对象1，然后将名字a绑定到这个对象上。在代码块中，名称的查找遵循LEGB规则（局部→ enclosing→全局→内置）。特别值得注意的是Python3.12引入的惰性求值机制，在处理类型别名和泛型时，只有在真正访问属性时才会进行求值，这极大地优化了处理相互递归类型时的性能。
# 执行模型
Python程序的运行经历了一个精密的流程：

1. **词法分析**：将源代码转换为标记流，识别缩进（INDENT/DEDENT）和关键字。
2. **语法分析**：将标记流在构建成抽象语法树（AST）。如果缩进不匹配，AST构建就会失败，导致语法错误
3. **编译**：将AST编译为字节码（.pyc文件）。
4. **解释执行**：Python虚拟机（PVM）逐行执行字节码。

在这个过程中，一场处理机制扮演着“中断与恢复”的角色。当运行时发生错误（例如：除以零），解释器就会引发异常，打断正常的控制流，跳转到except块执行。这种机制允许程序在面对错误时保持健壮，而不是直接崩溃。
